# 概念卡片 · Mullvad Exit IP Fingerprinting

> 12 张关键概念卡，覆盖从 WireGuard 协议、Mullvad 架构、到 RNG 副作用与 k-anonymity 的全链路理解。

---

## 卡 1 · WireGuard pubkey 作为身份

**一句话**：在 WireGuard 协议里，客户端的公钥 = 客户端的身份。没有用户名、没有密码、没有证书签发——服务器认识你的唯一方式是看你的公钥。

**关键属性**：

- 由客户端本地生成 (Curve25519)
- 自动注册到服务端
- 一旦泄露，撤销机制需要服务端手动配合
- 在 Mullvad 这种"按月付费"模型里，pubkey 和账号绑定

**为什么这关键**：本文揭示的攻击直接把 pubkey 用作了"种子源"——pubkey 就是 r。

---

## 卡 2 · Mullvad 出口 IP 池架构

**一句话**：Mullvad 用"少量服务器 + 每服务器多 IP 池"的反向架构，刻意制造 k-anonymity——希望同一台服务器后面有 k 个用户共享一个 IP 段，让单 IP 不足以指认。

**数据点**：

- 总服务器数：约 578
- 单服务器 IP 池：8 至 91 个 IP（实测 9 台样本）
- 总 IP 量：约 5 千到 1 万
- 同时活跃用户：约 100,000（行业估算）

**和 Proton VPN 的对比**：Proton 用 20,000+ 服务器，IP 几乎独占——每个 IP 后只有一两个用户，k=1 等于没有匿名。

---

## 卡 3 · 确定性 IP 分配（Deterministic Assignment）

**一句话**：Mullvad 不是每次连接都随机给你一个 IP——它根据你的 pubkey 计算出一个确定的"你应该用哪个 IP"，pubkey 不变 IP 就不变。

**为什么这样设计**：

- 避免用户在短时间内"跳"到不同 IP 上引起服务封禁
- 让流量 ratelimiting 在 IP 维度可预测
- 简化连接复用

**代价**：这个"确定性"如果对用户全局可观察，IP 本身就是身份。

---

## 卡 4 · WireGuard Key 轮换

**一句话**：Mullvad 官方客户端每 1-30 天自动轮换 pubkey（默认 72 小时）。第三方客户端**不轮换**。

**命令行查询**：

```
mullvad tunnel get
mullvad tunnel set rotation-interval <hours>
```

**关键事实**：用 `wg-quick up` 或 NetworkManager 直接配置 WireGuard 隧道，pubkey 永远不变——这种用户更容易被本文揭示的攻击命中。

---

## 卡 5 · IP Position 与 Pool Size 比例

**一句话**：把分给你的 IP 在池里的位置（1-based）除以池的总大小，得到一个 0-1 的比例——本文的核心观察是**所有服务器上这个比例都接近一样**。

**例**：

| 服务器 | 你拿到的 IP | Position | Pool size | Ratio |
|--------|-------------|----------|-----------|-------|
| au-syd-wg-101 | 103.136.147.53 | 49 | 60 | 0.816 |
| us-lax-wg-001 | 23.234.72.109 | 74 | 91 | 0.813 |
| za-jnb-wg-002 | 154.47.30.153 | 9 | 11 | 0.818 |

这三个值几乎相同——结论：服务器在用同一个 float r 决定 IP 索引。

---

## 卡 6 · 反直觉的 RNG 行为：`random_range(0..bound)`

**一句话**：在 Rust（以及大多数语言）里，"在 [0, bound) 内随机取整数"的实现是**取一个 [0, 1) 的 float，乘以 bound，取整**——改 bound 不消耗 entropy。

**核心后果**：

```rust
let mut rng = StdRng::seed_from_u64(seed);
for bound in 10..100 {
    let mut rng = StdRng::seed_from_u64(seed);  // reseed!
    let x = rng.random_range(0..bound);
    // x = floor(r * bound), r 永远是同一个
}
```

**为什么开发者会踩坑**：直觉认为"在 [0, 10) 抽数"和"在 [0, 11) 抽数"应该用掉不同 entropy。但实际上 RNG 只取一次 float，剩下的纯算术。

---

## 卡 7 · seed ratio = float r 区间

**一句话**：本文工具 [mullvad-seed-estimator](https://tmctmt.github.io/mullvad-seed-estimator/) 输入一个或多个 IP，反推出"你的底层浮点数 r 落在哪个区间"。

**区间宽度的含义**：

- 区间宽度 = 0.0034 → 0.34% 的所有用户在这个区间 → 100K 用户中 340 人 → 嫌疑池 340
- 加更多 IP 数据点 → 区间收窄 → 嫌疑池缩小
- 极限情况：3 个 IP 就能把嫌疑池缩到 50 人以下

---

## 卡 8 · k-Anonymity 与去匿名化

**定义**：在数据集中，每条记录至少与其他 k-1 条记录无法区分时，称这条记录满足 k-anonymity。

**Mullvad 的意图**：k = pool_size，比如美国洛杉矶 91 个 IP，每个 IP 后面有约 1000 个用户共享 → k ≈ 1000。

**实际结果**：因为所有服务器共享同一个 r，跨服务器的 k 退化到约 100K / 0.0034 ≈ 340。**有效 k 从 1000 降到 340**——更糟的是它**全局可观察**（任何观察过你两个 IP 的人都能算出你的 r 区间）。

---

## 卡 9 · 攻击模型一：同站点跨账号关联

**场景**：你是论坛 mod，怀疑某用户 B 是昨天封掉的 A 的小号。

**操作**：

1. 取 A 留下的 IP（比如 us-lax-wg-001 的某 IP）
2. 取 B 留下的 IP（比如 fi-hel-wg-201 的另一个 IP）
3. 把两者输入 estimator，分别得到 r 区间
4. 如果区间重合 → A = B 概率 > 99%

**前提**：两人都没在两次访问之间轮换过 pubkey。

---

## 卡 10 · 攻击模型二：跨站点身份合并

**场景**：你想知道某个匿名 Twitter 账号是不是某个匿名 GitHub 账号的同一个人。

**操作**：

- 通过法律或数据破解拿到两个站点的 IP 日志
- 提取 Mullvad 出口 IP（特征明显，IP 段可识别）
- 输入 estimator 比较 r 区间

**威胁等级**：高。这是过去的 VPN 模型里**几乎不可能**做到的能力。

---

## 卡 11 · 攻击模型三：预测性诱捕

**最阴险的一类**：r 是确定性函数，**可以从一次观察预测未来**。

**操作**：

- 取目标过去一次的 IP
- 计算 r
- **预测**目标在 Mullvad 其它服务器上未来会用什么 IP
- 在那些 IP 上部署监听 / 蜜罐 / 流量分析点

**为什么这是新颖威胁**：传统的 IP 监听是"听到了再分析"，本文使能"预测会在哪听到"。

---

## 卡 12 · 缓解与防御

**用户侧**：

- 强制轮换 pubkey（最简单：在 Mullvad app 里登出再登入）
- 不要在同一 pubkey 下连多台服务器
- 缩短自动轮换周期（默认 72 小时，可调低）

**Mullvad 侧（CEO 在 HN 评论里宣布）**：

- 已经在部分基础设施部署补丁
- 正在重新评估"哪些是有意行为、哪些应当关闭"
- 这是一个 trade-off：完全随机分配会破坏 ratelimiting；当前确定性又破坏 anonymity

**通用启示**：所有"按用户 hash 分配资源"的系统都需要审视——CDN 亲和、ratelimiting bucket、Sharding key——任何**确定性映射 + 全局可观察输出**的组合都是潜在指纹源。
